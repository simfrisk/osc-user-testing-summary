User Testing Insights & Customer Success Brief

üìä Executive Summary

Platform Purpose: OSC is a cloud platform enabling users to deploy and run open-source services without extensive technical knowledge. It targets developers and builders who need to implement open-source components quickly.

Based on feedback from 4 test users

üí≠ What Users Think OSC Is

User 1 Perspective

"A platform for running open source services on the cloud, aimed at being functional for both knowledgeable and novice users."

Use Case: Improving development environment, learning new technologies (especially Docker)

User 2 Perspective

"An open-source service that allows users to run servers easily."

Use Case: Assisting with regular tasks using open-source tools without deep expertise

User 3 Perspective

"An open source platform both for personal use but also for companies. You can upgrade your plan depending on what features you want to use."

Use Case: Small passion projects and building applications for companies

User 4 Perspective (Maryam)

"A website that offers open-source solutions and services."

Target Audience: Services are more oriented towards professionals and offices, which would help with making certain tasks more efficient

Use Case: To find solutions or services to suit specific needs

‚úÖ What Worked Well

Easy Wins

  • Service Creation: Users found starting services straightforward with clear, prominent buttons
  • Easy Install Option: The simplified installation path was accessible and intuitive - "required very little thinking" and "efficient to install Intercom using Easy Install"
  • No Cost Barrier: Free tier with 300 tokens removed fear of experimentation
  • Professional Design: Website appears professional and nicely designed
  • Docker Education: Services like Python Runner provided learning opportunities for containerization
  • Service Variety: Diverse offerings (Minecraft Server, Python Runner, Claude Code chatbot, Umami, Minio, Join Live) sparked curiosity
  • Confidence Building: Half of users weren't scared to try things out - interesting exploration encouraged experimentation
  • Search Functionality: The second search bar on Web Services page works well for finding services and searching by tags
  • Navigation: Generally easy to navigate once users understand the layout

‚ö†Ô∏è Critical Pain Points

Service Creation & Launch Issues

Issue: Service Creation Anxiety

  • Users wondered if they did something wrong while waiting for services (like Intercom) to finish creating
  • Initial thought that something broke when starting a running instance for the first time
  • Users started instances twice; on the third time it worked

Issue: Service Access Confusion

  • Users stuck on service login - didn't know settings were under "Service Parameters"
  • Unsure how to open instances - confused whether to click the card or the link
  • Difficulty "finding the actual application" after setup
  • Not obvious that Join Live's producer view requires "opening the application"

The Helper - A Missed Opportunity

"The helper could have solved a lot of their questions like login to Minio for example"
  • Users missed out on the Helper entirely during initial exploration
  • The Helper would have answered many of their questions (credentials, how to access services, etc.)
  • Users didn't know what to do with the Architect

Cost Anxiety & Token Transparency

"Fear of cost stopped them - they did not want to try out an expensive service so they would lose tokens"
  • Users afraid to experiment with services they perceived as "expensive"
  • Unclear what each service costs in tokens
  • Unaware of associated instances - didn't know services cost more if they don't delete all related instances
  • No clear indication of which services to delete together

Navigation & Discoverability

Issue: Duplicate Search Bars with Different Functionality

  • Web Services page has TWO search bars, both labeled "search service"
  • Top search bar: Cannot search for tags
  • Second search bar: Can search for tags
  • Users didn't realize the difference and thought both worked the same way
  • Confusing to have duplicate search functionality with different capabilities

Issue: Service Type Tags Not Interactive

  • Users wanted to click on service type tags to explore more similar services
  • Tags look clickable but aren't
  • Expected tag-based filtering/navigation

Issue: Star Rating Confusion

  • Users thought stars were GitHub stars but weren't sure
  • Unclear what the star metric represents

Issue: Navigation Structure Confusion

  • "Home", "Profile", and "Account" feel redundant and could be consolidated
  • Profile page only edits information that doesn't appear anywhere else - unclear value
  • Users confused about the difference between these sections
  • Must click logo explicitly to get home; "Open Cloud Source" text should also be clickable
  • Home button wouldn't work properly initially but worked later (intermittent issue)

Tutorial & Documentation Overwhelm

  • Users felt overwhelmed by too many tutorials
  • Tutorial titles were sort of hidden - hard to distinguish importance
  • Unclear which tutorials to start with
  • Documentation links disappear too quickly and aren't clickable

Technical Errors & Stability

  • Tenant Errors: If user didn't create a tenant, many errors occurred throughout the experience
  • Website Freezing: "Website stops working quite often" - everything becomes darker/grayed out
  • Website Not Responding: Sometimes the website wouldn't respond as expected
  • Favorites Bug: Cannot remove favorites once added - heart gets slightly gray but doesn't return to hollowed-out state
  • Disabled Features: Parts of website accessible even when not functional (e.g., when no team created)

UX Polish Issues

  • Loading Messages: "Loading..." text looks like a clickable button
  • Reload Service Page Button: Purpose unclear to new users
  • Inconsistent Experience: Different users saw different setup flows
  • Technical Jargon: Terms like "WHIP URL", "WHEP URL", "productions", "lines" confused users

üéØ Target Audience Insights

Who They Are

  • Developers & builders in businesses
  • People without deep DevOps expertise
  • Learners wanting to explore new technologies
  • Hobbyists working on passion projects
  • Those who need quick open-source deployments
  • People who don't want to "create everything from scratch"

What They Want

  • Easy setup without technical complexity
  • Clear instructions on how to use deployed services
  • Confidence to experiment safely
  • Cost transparency - know before they commit
  • Learning pathways for new technologies
  • Both personal and business use cases
  • Professional-looking platform they can trust

üí° CSM Priority Action Plan

üî• Immediate Critical Fixes (Week 1)

  • Fix Website Stability: Address the frequent freezing/graying out issue - this is critical for user trust
  • Fix Tenant Errors: Look up and resolve all errors that occur when users don't create a tenant
  • Add Service Creation Timer: Show approximate wait time during service creation to reduce anxiety
  • Fix Favorites Bug: Enable users to remove favorites once added
  • Post-Deployment Modal: Create popup with credentials, URLs, access instructions, and "Open Service" button

‚ö° High-Priority UX Improvements (Week 2-3)

  • Add Helper to Onboarding Tour: Make the Helper unmissable - it solves most user questions
  • Add Token Cost Display: Show cost on each service card and explain token consumption
  • Service Instance Packages: Group related instances and add "Delete All" option
  • Add "Open Service" Buttons: Clear buttons on service cards to launch services
  • Show Login Credentials on Cards: Add service parameters/credentials to service card UI
  • Add Start Button Delay: Prevent double-clicking on service start buttons
  • Make Service Tags Clickable: Enable filtering by clicking service type tags
  • Clarify GitHub Stars: Add "GitHub" to stars text

üìã Medium-Term Improvements (Month 1-2)

  • Improve Tutorial Organization: Create sections like "Start Here", "Main Tutorials", "Advanced Topics"
  • Consolidate Navigation: Merge or clarify Home/Profile/Account sections
  • Disable Inaccessible Features: Gray out features that require teams when no team exists
  • Better Error Messaging: Improve error page - explain what went wrong and next steps
  • Glossary Tooltips: Add hover tooltips for technical terms (WHIP URL, WHEP URL, productions, lines)
  • Visual Consistency: Fix "Loading..." appearance to not look like a button
  • Standardize Onboarding: Ensure all users see the same setup flows
  • Real-time Token Display: Show current balance and consumption rate
  • Make Logo Text Clickable: "Open Cloud Source" text should return to home

üéØ Customer Success Talking Points

  • Highlight the Helper: Proactively introduce users to the Helper - emphasize it answers most questions
  • Free Experimentation: Emphasize risk-free free tier BUT explain token costs upfront to reduce anxiety
  • Professional Platform: Highlight the professional design that users appreciated
  • Dual Use Cases: Position OSC for both personal passion projects AND business/company use
  • Learning Platform: Position as both deployment tool AND learning environment
  • Easy Install First: Always recommend Easy Install for new users
  • Set Cost Expectations: Explain token costs and associated instances during onboarding
  • Patience During Creation: Warn that service creation takes time (until timer is added)
  • Where to Find Things: Proactively explain where to find credentials, how to open services
  • Pre-Warn About Stability: Acknowledge and set expectations about occasional UI issues until fixed

üéì Overall Assessment

"Currently a little rough around the edges, but it works very smoothly once you know what you're doing."
"It looks like a nicely designed website. It looks professional."
"The helper could have solved a lot of their questions like login to Minio for example"

Core Strength: The platform successfully deploys services with minimal friction. The technology works, and the design appears professional.

Core Weaknesses:

  • Discovery Gap: The Helper exists but users miss it - it could solve most of their problems
  • Guidance Gap: Between "service deployed" and "user successfully using service" needs bridging
  • Transparency Gap: Cost anxiety prevents exploration - users need clear token information
  • Feedback Gap: Users unsure if things are working during waits - need progress indicators
  • Stability Issues: Technical errors and freezing hurt trust in the platform

User Confidence Varies: 2 of 4 users felt confident exploring independently, while 2 felt overwhelmed by terminology and costs. One user noted that "things aren't laid out plainly" which could intimidate less tech-savvy users. This suggests onboarding quality and clarity significantly impact user experience.

The Helper is Key: Multiple issues could be resolved by simply making the Helper more prominent in the user journey. This is a high-impact, relatively low-effort improvement.

CSM Focus Areas:

  • Introduce the Helper immediately - it's the secret weapon users are missing
  • Be transparent about costs upfront to reduce anxiety
  • Guide users through the "what happens after deployment" journey
  • Set expectations about service creation wait times
  • Explain associated instances and deletion workflows
  • Build confidence through clear, jargon-free communication
  • Manage expectations around stability issues until they're resolved
